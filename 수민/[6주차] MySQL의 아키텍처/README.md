## MySQL의 아키텍처

MySQL은 프로세스 기반의 프로그램이 아닌 스레드 기반의 프로그램이다.

### 1. 프로세스와 스레드의 차이

- **프로세스**
    
    OS에서 하나의 작업 단위
    
    하나의 프로세스는 그 자체로 프로그램이라 볼 수 있으며, 특정 목적을 위해 여러 역할의 프로세스가 묶여 하나의 프로그램을 구성하게 된다.
    
    - **장점**
    각자 할당받은 자원만 사용하므로 서로에게 영향을 미치지 않아 독립성과 안정성이 좋다.
    - **단점**
    스레드에 비하여 생성 종료 시에 많은 부하를 요구하며 컨텍스트 스위칭 비용이 많이 발생한다.
    
    *! 컨텍스트 스위칭: CPU가 다른 작업을 수행할 때 하던 작업의 상태(컨텍스트)를 홀드시켜 놓고 새 작업을 수행하는 것*
    
- **스레드**
    
    프로세스 안에서의 작업 단위
    
    프로세스의 하위 개념으로 여러 스레드는 프로세스가 할당 받은 메모리를 공유하며 작업을 수행하게 된다.
    
    - **장점**
    한 프로세스 안에서 메모리를 공유하기 때문에 프로세스 상태 및 메모리 맵핑이 불필요하여 상대적으로 컨텍스트 스위칭 비용이 적다.
    - **단점**
    한 프로세스 안에서 구성되어 있기 때문에 한 스레드에 문제가 생길 경우 다른 스레드에도 영향을 미칠 수 있다.

### 2. MySQL의 스레드 구조

foreground thread + background thread

- **foreground thread**
    - 사용자가 요청한 쿼리문장을 처리하는 스레드
    - 데이터 버퍼나 캐시 또는 직접 디스크나 인덱스 파일로 접근하여 데이터를 가져로는 스레드
- **background thread**
    - Insert buffer를 병합하는 스레드
    - 로그 기록 스레드
    - InnoDB 버퍼 풀의 데이터를 디스크로 기록하는 스레드
    - 데이터를 버퍼로 읽어들이는 스레드
    - 잠금, 데드락을 모니터링하는 스레드
    - 모든 백그라운드 스레드를 관리하는 메인 스레드

#### 메모리 구조

글로벌 메모리 영역 + 로컬 메모리 영역

*! 글로벌 메모리에는 기존에 저장된 데이터를 저장해두는 곳, 로컬 메모리는 그 데이터를 가지고 작업하는 곳이라고 생각하면 편하다*

- 글로벌 메모리 영역
    - 모든 스레드로부터 공유되는 영역
    - MySQL 서버를 실행할 때 운영체제로부터 설정한 만큼 할당받는 영역
    - Buffer Pool, Index Cache, Log Buffer 등..
- 로컬 메모리 영역
    - 클라이언트 스레드가 쿼리를 처리하는데 쓰이는 영역
    - 스레드 별로 독립적이라 절대 공유되지 않는 영역
    - 쿼리의 용도 별로 할당이 될 때도 있고 아예 할당되지 않을 때도 있다.
    - Soft Buffer, Join Buffer, Read Buffer 등..

### 3. MySQL의 엔진

MySQL 엔진 + 스토리지 엔진

#### MySQL 엔진

클라이언트로부터 오는 요청 처리 담당
SQL문장 분석, 최적화 등…

#### 스토리지 엔진

실제 데이터를 디스크 스토리지에 저장하거나 조회하는 부분 담당

MySQL의 스토리지 엔진 설정은 InnoDB가 default 값으로 되어있다. 기존에는 MyISAM 엔진을 사용하다가 transaction의 지원 부재, 더 높은 참조 무결성 제한, 더 높은 동시성 보장을 위해 InnoDB 엔진으로 전환되었다.

- **MyISAM**
예전의 MySQL 기본 엔진
    - transaction 없음
    - 외래키 없음
    - Crash 나면 복구 어려움
    - 테이블 단위 락
- **InnoDB**
5.5ver. 이후의 MySQL 기본 엔진
    - 트랜잭션 지원
    - 외래키 지원
    - Crash 복구 가능
    - 행(row) 단위 락
    - 데이터 무결성 보장

#### 핸들러 API

= 스토리지 엔진

MySQL 엔진의 쿼리 실행기에서 데이터를 쓰거나 읽어야 할 때 각 스토리지 엔진에 보내는 쓰기 또는 읽기 요청에 사용되는 API

### 4. 쿼리 실행 구조

1. 파서(Parser)가 들어온 쿼리 문장에 대해서 문법 확인을 하고 쿼리 문장을 적정한 단위로 분할하여 파서 트리를 만든다.
2. 전처리기(pre-processor)가 앞에서 만들어진 파서 트리를 보고 구조적으로 문제가 있는지 확인한다. (테이블 접근 권한 등)
3. 옵티마이저(Optimizer)가 요청받은 쿼리를 어떻게하면 가장 적은 비용으로 빠르게 즉, 최적으로 처리할 지를 결정한다.
    
    *! 앞으로 개발자가 해야할 일은 이 옵티마이저가 더 나은 결정을 할 수 있도록 쿼리를 작성하는 것이다. 옵티마이저 챕터가 따로 있으니까 별도로 읽어보자.* 
    
4. 실행 엔진은 옵티마이저에 의해 결정된 실행 계획대로 핸들러에게 지시한다.
5. 핸들러는 실행 엔진이 내려준 지시대로 작업을 수행한다.

### 5. 복제

2대 이상의 MySQL 서버가 동일한 데이터를 담도록 하는 실시간 동기화 기술

Master Server(읽기, 쓰기) + Slave Server(읽기)

마스터로 지정하는 방법은 바이너리 로그를 활성화하여 바이너리 로그가 기록되게 하는 것이고, 슬레이브로 지정하는 방법은 마스터 서버의 IP, PORT, 계정을 설정하는 것이다. 슬레이브에서 오는 동기화 요청에 대해 바이너리 로그를 통해 마스터는 처리가 가능하고, 슬레이브는 동기화한 내역을 릴레이 로그에 기록하여 데이터를 동기화한다.
